# SPEC: Trust And Evidence Model

## Status

Draft.

## Purpose

The trust and evidence model defines how Downstroke stores, verifies, rejects, quarantines, and retrieves project memory.

It prevents generated text, untrusted documents, bridge metadata, poisoned memory, stale facts, or tool outputs from becoming project truth without evidence.

## Core Rule

```txt
The LLM may propose memory.
Downstroke decides whether to persist it.
Evidence decides whether it can be verified.
Policy decides whether it can be used.
Doctor decides whether the system is healthy.
```

## Trust Levels

### trusted

Reserved for Downstroke-owned policy and internal generated metadata after validation.

Examples:

- Validated manifest defaults.
- Internal schema metadata.
- Internal repo fingerprint metadata.

### project

Project-owned files with stable source and hash.

Examples:

- `package.json`.
- `pnpm-lock.yaml`.
- `tsconfig.json`.
- Downstroke managed blocks.

### generated

Generated by an LLM or agent.

Examples:

- Proposed rule.
- Proposed plan.
- Proposed summary.
- Generated documentation.

### external

Produced by a tool, bridge, registry, MCP server, external document, remote API, or another agent system.

Examples:

- MCP tool description.
- CodeGraph output.
- BMAD output.
- Web document.
- Issue tracker content.

### untrusted

Unknown, user-provided, suspicious, or unresolved source.

Examples:

- Raw pasted content.
- Unknown file source.
- Retrieved document without provenance.
- Tool output with suspicious instructions.

## Status Values

### unknown

The fact is not established.

### inferred

The fact was derived by heuristic, LLM, or weak signal.

### observed

The fact was observed in a local source, but not verified by execution.

### verified

The fact has acceptable evidence.

### stale

The fact may no longer be true due to TTL, source changes, lockfile changes, git changes, or newer evidence.

### conflicted

Two or more facts disagree.

### quarantined

The fact or source is suspicious and cannot be injected into active context.

### rejected

The fact was reviewed or scanned and rejected.

## Trust Transition Rules

### Rule 1: LLM Output Cannot Verify

```txt
source.type = llm-output -> status != verified
```

### Rule 2: External Tool Output Cannot Verify Alone

```txt
source.type = external-tool -> status != verified unless local evidence exists
```

### Rule 3: Bridge Output Cannot Verify Alone

```txt
source.type = bridge -> status != verified unless local evidence exists
```

### Rule 4: User Input Cannot Weaken Security Policy Automatically

User input may request policy changes, but Downstroke must require explicit command path for security downgrades.

### Rule 5: Untrusted Cannot Become Trusted Directly

Blocked transition:

```txt
untrusted -> trusted
```

Allowed path:

```txt
untrusted -> quarantined -> observed -> verified
```

Only explicit review and evidence can move it forward.

### Rule 6: Stale Verified Facts Lose Active Authority

A stale fact may be shown as historical context but cannot be used as active truth.

## Verification Rules

### Typecheck Verified

A typecheck fact can be verified only if:

- Evidence type is `typecheck_report` or `command_exit_code`.
- Exit code is 0.
- Command is from allowlist or explicit policy.
- Evidence timestamp is not expired.

### Tests Verified

A test fact can be verified only if:

- Evidence type is `test_report` or `command_exit_code`.
- Exit code is 0.
- Test command is known.
- Evidence timestamp is not expired.

### Build Verified

A build fact can be verified only if:

- Evidence type is `build_report` or `command_exit_code`.
- Exit code is 0.
- Build command is known.
- Evidence timestamp is not expired.

### Stack Observed

A stack fact can be observed if it appears in project files.

Examples:

- `next` in `package.json`.
- `react-native` in `package.json`.
- `.csproj` file detected.
- `Program.cs` detected.
- `blazor` packages detected.

Stack observed is not equal to project functioning.

### Stack Verified

A stack fact can become verified only if a relevant command succeeds.

Examples:

- Next.js build passes.
- React Native typecheck passes.
- .NET build passes.
- Tests pass.

## Conflict Detection

Conflicts occur when two active facts with overlapping scope disagree.

Examples:

- Package manager detected as both `pnpm` and `npm`.
- Test runner detected as both `vitest` and `jest` without workspace separation.
- Build verified but latest build evidence failed.
- Bridge descriptor hash changed after approval.

Conflict handling:

1. Mark both facts `conflicted` or mark older fact `stale`.
2. Add risk record.
3. Exclude conflicted facts from active context truth.
4. Include conflict under `Open Risks`.

## Quarantine Rules

A fact or source enters quarantine if:

- Secret scanner fails.
- Prompt injection scanner fails.
- Source contains explicit instruction to override policy.
- Source contains suspicious hidden text patterns.
- Source origin is unknown and high impact.
- Bridge descriptor changed unexpectedly.
- External data requests tool execution.
- External data requests secret access.

Quarantine means:

- Not included in normal context packs.
- Not eligible for verification.
- Not allowed to influence policy.
- Only visible through inspect or doctor output.

## Memory Poisoning Controls

### Origin Binding

Every fact must keep original source type and source reference.

A summarized fact cannot erase the original origin.

Example:

```txt
external document -> LLM summary -> memory
```

The final memory record still includes original external source lineage.

### No Summary Laundering

LLM summaries cannot upgrade trust.

Blocked:

```txt
external -> llm-output -> trusted
```

Allowed:

```txt
external -> llm-output -> generated + external lineage
```

### Corroboration

High-impact facts need more than one signal.

Examples:

- Security policy changes.
- Bridge capability changes.
- Build verification.
- Dependency installation.
- Write permissions.

### Human Approval

Manual approval can accept a decision, but does not automatically prove technical truth.

Example:

A human can approve:

```txt
Use pnpm as default package manager.
```

A human cannot make this technically true without repo evidence:

```txt
All tests pass.
```

## Write Policy

Allowed by default:

```txt
.downstroke/**
new Downstroke-owned files
existing managed blocks
```

Blocked by default:

```txt
.env
secrets
lockfiles without explicit dependency intent
files outside repo root
files outside managed blocks
symlink escapes
arbitrary shell-generated patches
writes from quarantined context
```

## Doctor Checks

`doctor --experience` must check:

- Facts have source.
- Facts have trust level.
- Verified facts have evidence.
- Evidence files exist.
- Evidence has valid schema.
- Evidence logs are sanitized.
- No quarantined item appears in active context packs.
- No secret appears in facts or context packs.
- Stale facts are marked.
- Conflicted facts are not active truth.
- Bridge descriptors have expected hash if enabled.
- Nested repos are declared.

## Acceptance Criteria

- LLM-generated facts cannot be verified directly.
- External content cannot become instruction.
- Verified facts require evidence.
- Stale facts are excluded from active truth.
- Quarantined facts are excluded from context packs.
- Summary laundering is blocked.
- Bridge outputs start as external or untrusted.
- Doctor detects invalid trust transitions.
- Doctor detects verified facts without evidence.
- Doctor detects quarantine leakage.
